# Task: graceful math-notation handling in `sat-test-runner`

## Context

This app renders SAT practice tests from JSON files (schema below). Test JSON comes from
several sources — some hand-written, some generated by Claude, some generated by other
models. Those sources do not agree on how to encode math:

- Some use plain Unicode: `x² − 6x + 8 = 0`, `12π`, `√49`, `3/5`
- Some use LaTeX with `$...$` / `$$...$$` delimiters
- Some use LaTeX with `\(...\)` / `\[...\]` delimiters

Additionally, LaTeX arriving from other tools is frequently **corrupted**, because a single
backslash in a JSON string is an escape sequence. `"$\frac{1}{2}$"` does not contain `\frac`;
`JSON.parse` returns a form-feed character followed by `rac{1}{2}`. This parses without error
and renders as garbage. The same happens for `\times` (tab + `imes`), `\theta`, `\text`, `\to`,
`\binom`, `\beta`, `\bar`, `\rightarrow`, `\neq`.

Right now the app renders these strings as raw text, so LaTeX shows up as literal `$\frac{1}{2}$`
in front of students.

**Goal:** the app should render all three conventions correctly, repair what corruption it can,
and degrade visibly-but-safely on what it can't — without ever blanking out a question or
throwing during a timed section.

## Hard constraints

1. **No work on the timed critical path.** All normalization, repair, and validation happens
   once at import/load time. The normalized strings are what get stored in test state. During
   the timed window, rendering is a pure function of already-normalized strings. Do not
   normalize inside a render loop or a `useMemo` that runs per keystroke.
2. **A question must never fail to display.** Any error in math rendering degrades to showing
   the raw source in a visibly flagged `<code>` span. No blank stems, no error boundaries
   swallowing a question, no thrown exceptions escaping into the section runner.
3. **Existing tests must not regress.** `sat-practice-1.json` through `sat-practice-3.json` use
   Unicode-only notation with literal `$` for currency in word problems. These must render
   byte-identically to how they do today.
4. **Grid-in answer comparison stays raw-text.** Student input is compared against plain strings.
   Never run KaTeX or normalization on the comparison path.

## Deliverables

### 1. `lib/math/normalize.ts`

Pure functions, no React, fully unit-testable.

```ts
export type NormalizeResult = {
  text: string;              // normalized to \( \) / \[ \] convention
  repairs: Repair[];         // what was fixed
  problems: Problem[];       // what could not be fixed
};
export function normalizeMathText(input: string): NormalizeResult;
```

**Step A — repair mangled escapes.**

Build a repair table keyed on control character + following letters:

| Corrupted bytes | Intended |
|---|---|
| `\u000C` + `rac` | `\frac` |
| `\u000C` + `orall` | `\forall` |
| `\t` + `imes` | `\times` |
| `\t` + `heta` | `\theta` |
| `\t` + `ext` | `\text` |
| `\t` + `riangle` | `\triangle` |
| `\t` + `an` | `\tan` |
| `\t` + `o` (word boundary) | `\to` |
| `\b` + `inom` | `\binom` |
| `\b` + `eta` | `\beta` |
| `\b` + `ar` | `\bar` |
| `\r` + `ightarrow` | `\rightarrow` |
| `\n` + `eq` | `\neq` |

Apply the table. Then apply this general rule, which matters more than the table:

- **Inside a detected math span**, any remaining control character (`\f \b \v \r \t \n`) is
  definitely corruption. If it isn't in the table, record a `Problem` — do not guess.
- **Outside a math span**, only `\f \b \v \r` are corruption. Tab and newline are legitimate
  there (`\n\n` is the passage-break convention in the schema).

The `\n` → `\neq` entry is the one risky repair: only apply it when the match is inside a math
span, since `\n` followed by prose starting with "eq" is conceivable in ordinary text.

**Step B — convert `$$...$$` to `\[...\]`.** Unambiguous; `$$` is never currency.

**Step C — convert `$...$` to `\(...\)`, but only when the span is actually math.**

This is the delicate part. SAT word problems are full of literal dollar signs
("Five pencils cost $2… what is the cost of 35 pencils?"), and a naive pairing splitter will
swallow the intervening prose. Decide per candidate span using these rules, in order:

1. If a `$` is written as `\$`, it is literal. Unescape it and stop.
2. Count unescaped `$` in the string. **If the count is odd, treat every `$` in that string as
   currency** and convert nothing.
3. For each candidate span between paired delimiters, treat it as **math** only if *all* hold:
   - it is ≤ 60 characters
   - it contains no sentence-terminating punctuation followed by a space (`. `, `? `, `! `)
   - it matches `^[\s\w+\-*/=<>()\[\].,^_{}\\|:!'π√×÷≤≥≠°]+$`
   - it contains at least one of: `\`, `^`, `_`, `{`, or a single-letter variable token, or a
     relational operator (`=`, `<`, `>`, `≤`, `≥`, `≠`)
4. Otherwise treat it as **currency** — leave both delimiters as literal `$`.

When rule 4 fires on a string that also contains obvious LaTeX (`\frac`, `^`, `_`), record a
`Problem` rather than silently guessing. A human should look at that item.

**Step D — validate.** Extract every `\(...\)` and `\[...\]` span. Check delimiters balance.
Compile each span with `katex.renderToString({ throwOnError: true })`. Any failure becomes a
`Problem` carrying the span text and the KaTeX message. Also flag backslash-commands sitting
outside any math span — they will render as literal text.

### 2. `lib/math/MathText.tsx`

Client component. Renders an already-normalized string.

- Split on `\[...\]` (display) and `\(...\)` (inline).
- Text segments: HTML-escape first, then convert `\n` → `<br/>`. Order matters.
- Math segments: `katex.renderToString` with `output: "htmlAndMathml"` so screen readers get
  MathML, `strict: "warn"`.
- On a KaTeX throw, emit `<code class="math-error" title="…">{escaped raw}</code>`. Do not throw.
- Memoize on the input string. Import `katex/dist/katex.min.css` once, app-level.
- Props: `{ children: string; className?: string; block?: boolean }`. Default to `<span>`;
  render a `<div>` when `block` is set so display math isn't nested inside inline context.

Add a `.math-error` style: monospace, muted amber background, dotted underline. Visible to a
proctor, not alarming to a student.

### 3. Wire it into the runner

Replace raw string interpolation with `<MathText>` in:

- question stem (`block`)
- each answer choice (inline — see below)
- the review screen's explanation text
- section and test titles (cheap, and titles occasionally contain notation)

**Answer choices:** display math inside a choice breaks the option row layout. In
`normalizeMathText`, when the input is a choice, downgrade `\[...\]` to `\(...\)`. Add an
optional second argument `normalizeMathText(input, { context: "choice" })` to control this.

**Grid-ins:** normalize the *display* of the question stem, but the `answer` array stays
untouched and the comparison stays a raw trimmed case-insensitive string match. If a grid-in
`answer` value contains `$` or `\`, that is an authoring error — surface it as a `Problem` at
import time, because `\(8\)` will never equal `8`.

### 4. Import-time integration

Normalization runs in exactly two places:

- the built-in test loader that reads `public/tests/*.json`
- the "Import a test" paste/file handler

Both should call a single `normalizeTest(test): { test, report }` that walks the whole
structure. Store the normalized test in state.

Surface the report in the UI:

- **Zero problems:** no UI change.
- **Repairs only:** small dismissible note on the test-selection card — "N math expressions were
  auto-repaired." Do not block.
- **Problems present:** amber banner listing section + question number + the specific issue,
  with a "start anyway" affordance. Never hard-block a student from starting; a tutor may
  knowingly be running an imperfect file.

### 5. CLI gate

`scripts/validate-test.mjs <file>` — same checks, but exits non-zero on any problem. This is the
authoring-time gate; it should be stricter than the UI. It additionally scans the **raw file
text** before `JSON.parse`, because that is the only place single-backslash corruption is still
visible. Regex: `(?<!\\)\\([bfrt])(?=[a-zA-Z])`.

Add `"validate:test": "node scripts/validate-test.mjs"` to package.json scripts.

### 6. Tests

`lib/math/__tests__/normalize.test.ts` must cover at minimum:

- `"Five pencils cost $2. At this rate, what is the cost of 35 pencils?"` → unchanged, zero
  repairs, zero problems. **This is the most important test in the file.**
- `"A shirt costs $20 and a hat costs $15."` → unchanged (even number of `$`, but prose spans).
- `"If $x = 5$, what is $2x$?"` → `"If \\(x = 5\\), what is \\(2x\\)?"`
- `"$$ky - 4x = 7$$"` → `"\\[ky - 4x = 7\\]"`
- A string containing form-feed + `rac{1}{2}` → repaired to `\frac{1}{2}`, one `Repair` recorded.
- A string with an odd number of `$` → unchanged, zero conversions.
- `"\\(\\frac{1}{0\\)"` (unbalanced/invalid) → one `Problem`, and `MathText` still renders
  something rather than throwing.
- Existing `sat-practice-1/2/3.json` → zero repairs, zero problems. Assert this by loading the
  actual files, so it becomes a regression guard.

## Dependencies

`npm i katex` and `npm i -D @types/katex`. No `react-katex` — it adds a wrapper we don't need and
its error handling is less controllable than calling `renderToString` directly.

## Out of scope

Do not change the test JSON schema. Do not migrate the existing test files to LaTeX. Do not add
a math *input* method for grid-ins — students type plain numbers.

## Definition of done

- `npm run build` clean, no new type errors.
- All three existing tests render identically to before, including every `$` in the money word
  problems.
- A test file using `$...$` LaTeX renders correctly.
- A test file with single-backslash corruption either repairs cleanly or reports specific
  per-question problems, and is still fully takeable.
- No code path can throw during a timed section as a result of malformed notation.
